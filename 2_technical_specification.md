## 2. Назначение и цели

### 2.1. Назначение

Веб-приложение представляет собой интеллектуального чат-бота «АлИИна», встраиваемого на сайт мебельной студии. Бот принимает текстовые и голосовые обращения клиентов и отвечает в письменной форме в чате, выполняя функции онлайн-консультанта по заказу мебели.

### 2.2. Цели

1. Квалификация клиента и сбор контактных данных.
2. Расчёт ориентировочной стоимости мебели по алгоритму.
3. Предоставление справочной информации о компании, материалах, сроках, оплате и т.д.
4. Запись клиента на бесплатный замер.
5. Передача собранных данных менеджеру.

---

## 3. Целевая аудитория

- Посетители сайта `vstroy-mebel.ru`, заинтересованные в заказе мебели на заказ.
- География: Москва и Московская область.
- Устройства: десктоп, планшет, мобильные устройства.

## 4. Функциональные требования

### 4.1. Чат-интерфейс (Frontend)

| № | Требование | Описание |
|---|------------|----------|
| F-01 | Виджет чата | Всплывающее окно чата, встраиваемое на любую страницу сайта через `<script>`-тег или `<iframe>` |
| F-02 | Текстовый ввод | Поле ввода текстового сообщения с отправкой по Enter и кнопкой «Отправить» |
| F-03 | Голосовой ввод | Кнопка микрофона для записи голосового сообщения, конвертация речи в текст (Speech-to-Text) и отправка распознанного текста боту |
| F-04 | Отображение сообщений | Лента сообщений с визуальным разделением сообщений клиента и бота (аватары, цвета, выравнивание) |
| F-05 | Приём файлов | Возможность отправки фото/файлов проекта (чертежей) клиентом |
| F-06 | Отображение ссылок | Корректное распознавание и отображение ссылок в сообщениях |
| F-07 | Индикатор набора | Анимация «бот печатает…» во время ожидания ответа |
| F-08 | Адаптивный дизайн | Корректное отображение на десктопе (≥1024px), планшете (768–1023px) и мобильных устройствах (<768px) |
| F-09 | Звуковое уведомление | Звуковой сигнал при получении нового сообщения от бота, если окно чата свёрнуто |
| F-10 | Сворачивание/разворачивание | Кнопка для сворачивания чата в иконку и обратно |

### 4.2. Логика бота (Backend / AI-ядро)

| № | Требование | Описание |
|---|------------|----------|
| B-01 | Приветствие | Однократное представление: «Я — АлИИна, ваша помощница и консультант Мебельной Студии "Встрой-Мебель.ру"» |
| B-02 | Запрос имени | Обязательный запрос имени клиента перед продолжением консультации. Повторный запрос при отсутствии ответа |
| B-03 | Обращение по имени | Обращение к клиенту по имени и на «вы» на протяжении всей беседы |
| B-04 | Тон общения | Короткие, вежливые, тактичные ответы. Редкое использование эмодзи |
| B-05 | Сбор данных в JSON | Внутреннее хранение данных сессии (`Client_name`, `Mebel_name`, `Dlina`, `Gorod`, `Ylitsa`, `Vremya_client`, `Client_Project`, `Orientir_price`). JSON клиенту не показывать |
| B-06 | Поочерёдный сбор | Если нужно более 2 данных — запрашивать строго по одному |
| B-07 | Уточнение категории мебели | При обобщённых запросах («хочу мебель», «хочу шкаф») — уточнять категорию из списка |
| B-08 | Проверка исключений | Проверка по базе [DB-EXCEPTION]. Если мебель в списке исключений — сообщить, что не занимаемся |
| B-09 | Вопрос перед расчётом | Спросить (один раз за беседу): модель с сайта или расчёт по своим размерам |
| B-10 | Подтверждение перед расчётом | Обязательное подтверждение: «Подтвердите, что рассчитываем именно эту мебель» |
| B-11 | Расчёт по категории | Формула: `Цена категории × Длина (м)` по базе [DB-PRICE-CATEGORY] |
| B-12 | Расчёт по номеру модели | Формула: `Цена модели × Длина (м)` по базе [DB-MODEL-NUMBERS] |
| B-13 | Конвертация единиц | Автоматический перевод см → м, мм → м. Парсинг формата `159×60×240` → длина = 1.59 м |
| B-14 | Формат ответа по цене | Шаблон: «Ориентировочная стоимость — XXXXX рублей…» без показа формулы расчёта |
| B-15 | Алгоритм для гардеробных | Запрос ширины, глубины, формы (П/Г/Прямая). Расчёт: П = `(глубина×2)+ширина`, Г = `глубина+ширина`, Прямая = `ширина` |
| B-16 | Алгоритм для угловых шкафов | Запрос двух сторон. Расчёт: `Ширина №1 + Ширина №2` |
| B-17 | Алгоритм для угловых шкафов-купе | Запрос двух сторон. Расчёт: `Ширина №1 + Ширина №2` |
| B-18 | Обработка ссылок | Извлечение названия и номера модели из URL вида `...uglovoi-shkaf-s-zerkalom-23`, поиск по базе |
| B-19 | Работа с проектом клиента | При получении файла — шаблонный ответ, запрос времени и телефона, передача менеджеру, остановка диалога |
| B-20 | Вызов менеджера | При запросе менеджера — сбор телефона и времени, передача данных, остановка диалога |
| B-21 | Предложение замера | После каждого расчёта — предложить записаться на бесплатный замер |
| B-22 | Сбор адреса для замера | Поочерёдный запрос: город → улица → телефон |
| B-23 | Завершение диалога | Спросить, есть ли ещё вопросы. Если нет — поблагодарить и остановить беседу |
| B-24 | Ограничения ответов | Не сообщать цену за метр, стоимость полок/ящиков/фурнитуры. Не считать без подтверждения/размеров/при наличии проекта |
| B-25 | Информационные запросы | Ответы по базе [DB-COMPANY-INFO]: фурнитура, материалы, сроки, история, адрес, график, скидки, гарантии и т.д. |

### 4.3. Управление базами данных

| № | Требование | Описание |
|---|------------|----------|
| D-01 | База [DB-MODEL-NUMBERS] | Хранение моделей мебели: номер, название, цена. Возможность добавления, редактирования и удаления записей |
| D-02 | База [DB-PRICE-CATEGORY] | Хранение категорий мебели и цен за метр. Возможность обновления цен |
| D-03 | База [DB-COMPANY-INFO] | Хранение справочной информации о компании. Возможность редактирования текстовых блоков |
| D-04 | База [DB-EXCEPTION] | Хранение списка мебели, которой компания не занимается. Возможность обновления |
| D-05 | Административная панель | Веб-интерфейс для менеджера/администратора для управления всеми базами без необходимости редактирования кода |
| D-06 | Быстрое обновление | Изменения в базах вступают в силу мгновенно, без перезапуска приложения |

### 4.4. Передача данных менеджеру

| № | Требование | Описание |
|---|------------|----------|
| M-01 | Сохранение диалогов | Все диалоги сохраняются с временными метками |
| M-02 | Экспорт JSON клиента | Собранные данные клиента доступны менеджеру в админ-панели |
| M-03 | Уведомления | Уведомление менеджера (email / Telegram) при: передаче проекта, запросе менеджера, записи на замер |

## 8. Пользовательские сценарии

### 8.1. Сценарий: Расчёт стоимости по категории

```
1. Клиент открывает чат → бот представляется
2. Бот спрашивает имя → клиент отвечает
3. Клиент: «Хочу шкаф-купе»
4. Бот (один раз): «Понравилась ли модель на сайте или считаем по вашим размерам?»
5. Клиент: «По моим размерам»
6. Бот: «Подтвердите, что рассчитываем шкаф-купе»
7. Клиент: «Да»
8. Бот: «Какая ширина (длина) шкафа?»
9. Клиент: «200 см»
10. Бот: «Ориентировочная стоимость — 120 000 рублей...»
11. Бот: «Я запишу вас на замер...»
```

### 8.2. Сценарий: Расчёт по номеру модели

```
1. Знакомство (шаги 1–2 как выше)
2. Клиент: «Мне нравится модель №15»
3. Бот: «Подтвердите, что рассчитываем шкаф-купе с графитовыми зеркалами (№15)»
4. Клиент: «Да, длина 1.8 м»
5. Бот: «Ориентировочная стоимость — 95 760 рублей...»
6. Бот предлагает замер
```

### 8.3. Сценарий: Клиент прислал проект

```
1. Знакомство (шаги 1–2)
2. Клиент отправляет файл чертежа
3. Бот: шаблонное сообщение с запросом времени и телефона
4. Клиент даёт данные
5. Бот: «Всю информацию передам менеджеру»
6. Бот завершает диалог
```

### 8.4. Сценарий: Гардеробная (специальный алгоритм)

```
1. Знакомство → клиент хочет гардеробную
2. Бот: «Какая ширина (длина)?» → 3 м
3. Бот: «Какая глубина?» → 1.5 м
4. Бот: «Какая форма: П, Г или Прямая?» → П
5. Расчёт: (1.5 × 2) + 3 = 6 м × 29 000 = 174 000 ₽
6. Бот: «Ориентировочная стоимость — 174 000 рублей...»
```

### 8.5. Сценарий: Информационный запрос

```
1. Знакомство
2. Клиент: «Какие материалы используете?»
3. Бот отвечает из [DB-COMPANY-INFO] — информация о ЛДСП Egger, МДФ, фурнитуре и т.д.
```

### 8.6. Сценарий: Запрос вне компетенции

```
1. Знакомство
2. Клиент: «Сколько стоит отдельная полка?»
3. Бот: «К сожалению, я не владею информацией о стоимости отдельных полок.
   Эту информацию уточнит менеджер после замера.»
```

## 9. Интерфейс

### 9.1. Виджет чата

- **Кнопка-триггер:** круглая иконка в правом нижнем углу экрана (фирменный цвет компании).
- **Окно чата:** выезжающая панель ~400×550px (десктоп), полноэкранная (мобильные).
- **Шапка:** логотип + название «АлИИна — Консультант», кнопка закрытия.
- **Лента сообщений:** сообщения бота слева (с аватаром), клиента справа.
- **Нижняя панель:** поле ввода текста, кнопка отправки, кнопка микрофона, кнопка прикрепления файла.

### 9.2. Административная панель

- **Авторизация:** логин/пароль.
- **Раздел «Базы данных»:** CRUD-интерфейс для каждой из 4 баз (DB-MODEL-NUMBERS, DB-PRICE-CATEGORY, DB-COMPANY-INFO, DB-EXCEPTION).
- **Раздел «Диалоги»:** список всех диалогов с фильтрацией по дате, имени клиента. Просмотр полного диалога и собранного JSON.
- **Раздел «Замеры»:** список заявок на замер (город, улица, телефон, время).

## 10. Системный промт для ИИ

Системный промт формируется автоматически на основе:
1. **Алгоритма работы** — части 1–12 из `1_idea.md` (роль, правила, ограничения, алгоритмы расчёта).
2. **Актуальных баз данных** — содержимое JSON-файлов DB-MODEL-NUMBERS, DB-PRICE-CATEGORY, DB-COMPANY-INFO, DB-EXCEPTION подставляется в промт при каждом обращении.

Это обеспечивает:
- Мгновенное обновление информации при изменении баз через админ-панель.
- Единый источник истины для поведения бота.

| Бот не показывает JSON, формулы, цену за метр | ✅ |
| Бот отклоняет запросы вне компетенции | ✅ |

---

## 13. Риски и ограничения

| Риск | Описание | Митигация |
|------|----------|-----------|
| Стоимость API OpenAI | Каждое обращение к GPT-4 стоит денег | Кэширование частых ответов, ограничение длины контекста |
| Неточность ИИ | Бот может дать некорректный расчёт | Двойная проверка формул в промте, логирование всех расчётов |
| Web Speech API | Может не работать в некоторых браузерах | Fallback на Whisper API или сообщение о несовместимости |
| Объём промта | Большие базы данных увеличивают промт и стоимость токенов | Оптимизация формата данных, при необходимости — RAG |

## 14. Глоссарий

| Термин | Определение |
|--------|------------|
| Виджет чата | Встраиваемый на сайт компонент чата |
| Системный промт | Инструкция для ИИ-модели, определяющая поведение бота |
| JSON клиента | Внутренняя структура данных, собранных в ходе диалога |
| Замер | Бесплатный выезд мастера для обмера помещения |
| RAG | Retrieval-Augmented Generation — метод подгрузки данных в контекст ИИ |
